# پنل مدیریتی تبچی — فاز ۱

ربات تلگرامی **جدا و مستقل** برای مدیریت مشتری‌ها و پلن‌ها. کد و دیتابیس این پنل کاملاً
از پروژه‌ی اصلی تبچی جداست (جداول `tenants`, `plans`, `audit_log`, `panel_state`) و فقط
از همون MySQL که تبچی روش دیپلوی شده استفاده می‌کنه — بدون هیچ تغییری در جداول تبچی.

## دیپلوی (سرویس دوم و مستقل روی Railway)

1. یه سرویس جدید توی همون پروژه‌ی Railway بساز (یا یه پروژه‌ی جدا) و همین پوشه‌ی
   `admin_panel/` رو به‌عنوان root سرویس تنظیم کن.
2. متغیرهای محیطی `.env.example` رو ست کن:
   - `PANEL_BOT_TOKEN`: توکن یه ربات تلگرامی **جدید** (از BotFather) — جدا از ربات اصلی تبچی
   - `PANEL_OWNER_ID`: آیدی عددی تلگرام خودت (فقط همین آیدی به پنل دسترسی داره)
   - `API_ID` / `API_HASH`: همون مقادیر پروژه‌ی اصلی
   - متغیرهای `MYSQL*`: اگه همون دیتابیس Railway رو استفاده می‌کنی، این متغیرها رو از سرویس
     MySQL موجود Reference بده (نیازی به دیتابیس جدید نیست)
3. اجرا: `python main.py` (طبق Procfile)

## چیزی که فاز ۱ داره

- `/start` (فقط برای `PANEL_OWNER_ID`) → داشبورد: تعداد مشتری‌ها به تفکیک وضعیت
- **مدیریت مشتری‌ها**: افزودن دستی (آیدی تلگرام + اسم اختیاری)، مشاهده‌ی جزئیات، تمدید
  دستی اشتراک، تغییر پلن، فعال/مسدودسازی
- **مدیریت پلن‌ها**: افزودن پلن (نام/قیمت/مدت/سقف‌اکانت/سقف‌لایه)، ویرایش هرکدوم جدا،
  فعال/غیرفعال‌سازی — همه‌چیز تو دیتابیسه، نیازی به دیپلوی جدید برای تغییر پلن نیست
- هر عملیات حساس (تمدید، مسدودسازی، تغییر پلن، ویرایش پلن) در جدول `audit_log` ثبت می‌شه

## چیزی که فاز ۲ اضافه کرد

تبچی (پروژه‌ی اصلی) الان چندمستأجریه — هر کاربر تلگرام با `/start` فضای مستقل خودش رو
می‌گیره. این پنل همچنان جدا و فقط برای owner می‌مونه.

## چیزی که فاز ۳ اضافه کرد (پرداخت + فعال‌سازی خودکار)

- **کارت‌به‌کارت دستی**: مشتری تو ربات تبچی رسید رو می‌فرسته → مستقیم به owner (تو همون
  ربات تبچی) اطلاع داده می‌شه → owner از همینجا (💳 پرداخت‌های در انتظار) تایید/رد می‌کنه
  → تایید = تمدید خودکار اشتراک + پیام به مشتری.
- **زرین‌پال**: مشتری تو ربات تبچی پلن رو انتخاب می‌کنه → لینک پرداخت می‌گیره → بعد از
  پرداخت، این سرویس (`admin_panel`) از طریق وب‌سرور داخلیش (`/zarinpal/callback`) پرداخت
  رو تایید و اشتراک رو **خودکار** فعال می‌کنه.
- به همین خاطر Procfile این سرویس از `worker` به `web` تغییر کرد — Railway بهش یه دامنه‌ی
  عمومی می‌ده که باید تو تنظیمات زرین‌پال (`callback_url`) و env تبچی
  (`ZARINPAL_CALLBACK_URL`) استفاده بشه: `https://<این-دامنه>/zarinpal/callback`
- هر تننت جدید ۳ روز آزمایشی رایگان می‌گیره (`TRIAL_DAYS_DEFAULT` در `database.py`)؛
  بعد از تموم‌شدن، تبچی خودش رو قفل می‌کنه (داده‌ها می‌مونن) تا تمدید بشه.
- یادآوری خودکار انقضا (۳ روز قبل / ۱ روز قبل / روز انقضا) از طریق ربات تبچی — در
  `workers/expiry_reminder.py`.

### متغیرهای env جدید لازم برای فاز ۳
- تو `admin_panel/.env`: `TABCI_BOT_TOKEN`, `TABCI_BOT_USERNAME`, `ZARINPAL_MERCHANT_ID`
- تو `.env` تبچی: `PANEL_OWNER_ID`, `CARD_NUMBER`, `CARD_HOLDER`, `ZARINPAL_MERCHANT_ID`,
  `ZARINPAL_CALLBACK_URL`

## چیزی که هنوز نداره (عمداً)

- تیکتینگ/broadcast (فاز ۵)
- کد تخفیف/آزمایشی قابل صدور (اگه لازم شد جدا اضافه می‌کنیم)

## فاز ۴ (محدودیت منابع + هشدار مصرف)

- هر پلن از قبل `max_accounts` و `max_layers` داشت (فاز ۱)؛ حالا واقعاً اعمال می‌شن:
  - `/add_account` قبل از شروع فلوی ورود چک می‌کنه اکانت‌های فعال تننت به سقف پلنش رسیده یا نه.
  - ساخت لایه‌ی جدید هم همین‌طور چک سقف `max_layers` می‌شه.
  - تننت‌های بدون پلن (هنوز تو آزمایشی) سقف پیش‌فرض می‌گیرن: `TRIAL_MAX_ACCOUNTS`/`TRIAL_MAX_LAYERS`
    در `utils.py` تبچی (پیش‌فرض هرکدوم ۱).
  - `0` تو ستون‌های پلن یعنی نامحدود (بدون تغییر نسبت به فاز ۱).
- داشبورد پنل حالا **تعداد کل اکانت‌های فعال روی کل سرور** رو نشون می‌ده، و اگه
  `MAX_TOTAL_ACCOUNTS` رو ست کنی، درصد مصرف + هشدار رنگی (۷۵٪/۹۰٪) هم نشون می‌ده.
- دکمه‌ی جدید «📊 پرمصرف‌ترین مشتری‌ها» — لیست مشتری‌ها به ترتیب تعداد اکانت فعال.
- جزئیات هر مشتری حالا مصرف واقعیش (اکانت فعال / سقف پلن) رو هم نشون می‌ده.
- `capacity_monitor.py` (worker جدید تو `main.py` پنل): هر ۳۰ دقیقه چک می‌کنه و وقتی مصرف
  کل از ۷۵٪ یا ۹۰٪ `MAX_TOTAL_ACCOUNTS` رد بشه، یه‌بار (نه هر بار) به owner پیام می‌ده.

### متغیر env جدید لازم برای فاز ۴
- تو `admin_panel/.env`: `MAX_TOTAL_ACCOUNTS` (اختیاری؛ 0 یعنی بدون هشدار ظرفیت)

## فاز ۵ (پشتیبانی/تیکت + Broadcast) — آخرین فاز

- **تیکت پشتیبانی:** دکمه‌ی «🆘 پشتیبانی» تو منوی اصلی ربات Tabchiz (حتی برای مشتری با
  اشتراک تموم‌شده هم کار می‌کنه). پیام مشتری تو جدول `tickets` ثبت و مستقیم برای owner
  فرستاده می‌شه. پنل مدیریتی → «🎫 تیکت‌های پشتیبانی» → پاسخ یا بستن بدون پاسخ؛ پاسخ از
  طریق `notify_tenant` (توکن ربات Tabchiz) مستقیم برای مشتری ارسال می‌شه.
- **اطلاع‌رسانی همگانی:** پنل مدیریتی → «📢 اطلاع‌رسانی همگانی» → انتخاب مقصد (همه یا فقط
  یه پلن خاص) → نوشتن متن → پیش‌نمایش و تایید نهایی → ارسال با فاصله‌ی کوتاه بین پیام‌ها
  (جلوگیری از فلود) + گزارش تعداد موفق/ناموفق در پایان.

### تغییر نام برند
- نام نمایشی محصول از «تبچی» به **Tabchiz | تبچیز (نسخه آزمایشی)** تغییر کرد (پیام خوش‌آمد،
  دکمه‌ها، داشبورد پنل). توجه: نام و عکس پروفایل خودِ ربات تلگرام باید جدا از BotFather تغییر
  کنه؛ این تغییر کد فقط متن‌های داخل چت رو پوشش می‌ده.
